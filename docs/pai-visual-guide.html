deep project guide

How the whole project fits together

pai is not trying to be a broad personal AI shell. Its center of gravity is the recurring decision loop: Ask -> Program -> Brief -> Action or Correction -> Next Brief improves. This guide explains the complete implementation around that loop, including chat, memory, knowledge, background execution, research, swarm analysis, Telegram delivery, and validation.

product model

Primary nouns, internal tables, and route surfaces

The project intentionally separates product language from some of the underlying table names. This is one of the most important things to understand: several product objects are adapters over older or more implementation-oriented storage.

Program = the watch

Long-lived, recurring, decision-oriented

A Program is the ongoing thing the system keeps evaluating over time. It survives many runs and many briefs.

  • Question it answers: "What are we continuing to watch or manage?"
  • Lifespan: days, weeks, or months
  • Output: repeated briefs as the situation changes
  • Example: "Keep watching Atlas launch readiness"
Action = the move

Short-lived, bounded, executable follow-through

An Action is a concrete next step. It should be smaller than the Program, and it should be possible to complete, reopen, or cancel it.

  • Question it answers: "What exactly should happen next?"
  • Lifespan: one bounded follow-through step
  • Output: status changes, not new briefs
  • Example: "Ask infra lead whether blocker X clears by Friday"

Program and Action should not sound the same

If an Action merely repeats the Program, it is probably too vague. A weak Action would be "keep watching Atlas launch readiness" because that is still the Program. A stronger Action would be "message the infra lead for blocker status before Friday's review" because it is a specific piece of follow-through generated from the latest brief.

The simplest rule: a Program creates recurring evaluation; an Action creates accountable movement.

Program

A Program is an adapter over scheduled_jobs. It carries both authored state like objective, phase, delivery mode, source refs, preferences, constraints, and open questions, and runtime state like latest brief, last delivered time, and last signal hash. It is the recurring watch, not the immediate next step.

Action

An Action is stored as a row in tasks. User-facing surfaces now treat it as optional tracked follow-through or a recommended move, while the storage and route names remain tasks and /api/tasks for compatibility. A good Action is narrower than the Program and only exists when there is a real manual step worth revisiting.

Brief Contract

All meaningful briefs are converging on the same schema: recommendation, what changed, evidence, memory assumptions, next actions, and correction hook. Research and swarm outputs are normalized into this shape before delivery.

Implementation anchors for the noun model
  • Program adapter and runtime state: packages/plugin-schedules/src/schedules.ts
  • Program enrichment and history joins: packages/server/src/routes/programs.ts
  • Action/task schema and source linkage: packages/plugin-tasks/src/tasks.ts
  • Belief schema, provenance, corrections: packages/core/src/memory/memory.ts
  • Knowledge source store and chunk search: packages/core/src/knowledge.ts
outside-in view

Primary user flows

These are the flows the product is built around. They are shown first from the user point of view, then later from the internal runtime point of view.

Flow A: Ask for help right now

User

Types a question in web chat or Telegram.

->
Assistant

Runs the chat pipeline, tools, and memory retrieval to answer in the current thread.

Assistant

May recall beliefs, query knowledge, search the web, or manipulate tasks.

->
User

Gets a response immediately, even if no Program is created.

Flow B: Keep watching this

User

Says "keep watching this" or asks for ongoing follow-through.

->
Program create path

Assistant or UI creates a Program tied to thread/chat and desired cadence.

Program

Waits for due cadence and background execution.

->
Brief

New research or analysis is transformed into a recommendation-first brief.

Flow C: Brief -> Action

Brief

Shows recommended next actions in the Inbox or Telegram.

->
User

Accepts a next step and turns it into a tracked Action.

Action

Lives in the task store with source linkage back to the brief or program, but remains a smaller, executable step.

->
Future brief

Open, stale, or completed actions shape later recommendations.

Program

Still remains the long-running watch in the background.

->
Action does not replace it

The Action operationalizes one move inside the Program rather than rephrasing the Program itself.

Flow D: Brief -> Correction

User

Notices a wrong assumption or outdated memory in the brief.

->
Correction route

Replaces the old belief, preserves history, and invalidates the old assumption.

Belief graph

Supersession and provenance are updated immediately.

->
Future brief

The next recommendation uses the replacement belief instead of the invalidated one.

Flow E: Learn a page or document

User

Uploads a file, shares a URL, or adds a source to learn.

->
Knowledge store

Content is chunked, embedded, and stored as reusable source material.

Knowledge

Supports future answers, research runs, and background learning.

->
Beliefs, sometimes

Only high-signal personal facts may later be extracted into beliefs by background learning.

Flow F: Telegram companion

Telegram

Receives pushed briefs linked to thread, program, or owner chat.

->
User

Can reply, create an action, complete an action, or correct a belief directly from Telegram.

Shared backend

No Telegram-only shadow model exists for Programs, Briefs, Actions, or Beliefs.

->
Same loop

Telegram is a companion surface, not a separate product model.

What the user actually sees today
  • Programs have their own page and history view.
  • Briefs are delivered in the Inbox and optionally pushed to Telegram.
  • Recommended moves are visible in Brief detail and Programs can open source-scoped follow-through, while the dedicated route still remains /tasks internally.
  • Memory has explicit correction surfaces for beliefs.
  • Knowledge has its own source list, upload, learn, search, crawl, reindex, and delete routes.
inside-out view

Internal dataflows and runtime mechanics

The following sections show what the runtime is actually doing under the user-facing flows. Each subsection names the dominant reads, writes, and async edges.

Chat dataflow

Thread history is loaded, tools run, responses stream or send, and fire-and-forget post-processing extracts durable memory. Chat is how many flows start, but it is not the primary long-term product object.

Program dataflow

Programs are created from chat or UI, stored as scheduled jobs, and later picked up by the worker loop for research or swarm execution. Program runtime state records whether anything materially changed.

Brief dataflow

Daily, research, and swarm runs all converge on a standard brief contract. The same shape makes it possible to unify delivery, action extraction, correction, and validation.

1. Web or Telegram chat request

Surface request

User message enters web chat route or Telegram chat handler.

->
Thread continuity

The system gets or creates a thread, loads recent messages, and applies a thread lock.

Tool-enabled model call

Assistant can call memory, knowledge, tasks, web search, Program creation, or report tools.

->
Message persistence

User, tool summaries, and assistant responses are appended back to the thread history.

After-response hooks

Post-processing may extract confirmed memory and episodic summaries.

->
Future context

Later chats and briefs can retrieve those beliefs and episodes.

2. Program runtime and scheduling

Program creation

Stored in scheduled_jobs with question, cadence, authored state, and thread/chat linkage.

->
WorkerLoop

Polls due schedules and avoids starting duplicate queued/running work for the same Program.

BackgroundDispatcher

Queues research, swarm, and daily briefing work with source priority and traffic awareness.

->
Execution engine

Program execution mode decides whether the run uses lighter research or deeper swarm analysis.

Program runtime state

latestBriefId, lastDeliveredAt, lastEvaluatedAt, and lastSignalHash update after evaluation.

->
Change-gated delivery

If nothing materially changed, a Program may be evaluated without creating a user-visible new brief.

3. Daily brief generation

Context assembly

Reads open tasks, recently completed tasks, active goals, active Programs, recent beliefs, episodes, and knowledge source metadata.

->
Raw context object

This becomes both the prompt context and the deterministic fallback context.

LLM or fallback

The system tries to generate the standard brief shape. If the model is unhealthy or output is malformed, a deterministic brief is produced instead.

->
Brief row write

The result is stored into briefings with type daily.

Action-aware merge

Existing linked actions can reshape recommendation and next actions so the brief does not repeat already-completed work.

->
Inbox delivery

The latest ready daily brief becomes visible in the Inbox and may later be pushed to Telegram.

4. Research and swarm report normalization

Job completion

Research or swarm produces a report, structured result, render spec, and optional visuals.

->
Standard brief conversion

buildReportBriefSection() turns report outputs into recommendation-first brief sections.

Action summary

Open and stale linked actions are fed into the brief builder.

->
Signal hash

A normalized brief hash drives change-gated suppression when a Program is configured that way.

Appendix and knowledge

The long report stays available through brief appendix fields and is also learned back into the knowledge base.

->
Future reuse

Later research, swarm, and knowledge queries can build on prior report output.

5. Action lifecycle

Action creation

User creates an Action from brief detail, Program view, or Telegram command.

->
Task row

Stored in tasks with source_type and source_id for linkage.

Status changes

Completion or reopen updates status and timestamps.

->
Product signals

Brief action created and completed events are recorded for measurement.

Future brief context

Open, stale, and recently completed actions are re-read into daily and Program-based brief generation.

->
Behavior change

Later briefs can say "close the stale action first" or avoid repeating already finished follow-through.

6. Belief lifecycle and correction

Belief creation

Comes from user memory writes, chat extraction, or background learning.

->
Trust metadata

Beliefs carry origin, freshness, correction state, sensitivity, provenance, and supersession links.

Correction

The old active belief is invalidated, a new replacement belief is created, and both history and provenance are preserved.

->
Immediate suppression

Future recommendation logic reads active beliefs, so the invalidated assumption stops influencing later briefs.

Recall

Beliefs are semantically ranked with importance, recency, stability, and subject-aware scoring.

->
Context retrieval

Beliefs and episodes form the high-signal personal context layer.

7. Knowledge ingestion and retrieval

Ingestion

URL learning, uploaded docs, crawled sub-pages, chat attachments, and completed reports become knowledge sources.

->
Chunk store

Content is split into coherent chunks, contextualized with title/domain, and embedded when possible.

Querying

Search combines FTS, source-title/tag matching, semantic ranking, diversity limits, and freshness decay.

->
Downstream use

Assistant tools, research, swarm, and unified retrieval all consume the knowledge base.

Lifecycle controls

Sources can be retagged, reindexed, crawled, deleted, or expired via TTL cleanup.

->
Controlled corpus

Knowledge stays reusable without becoming an unbounded stale source dump.

8. Background learning

Signal gathering

New threads, completed research, completed tasks, and new knowledge sources are collected since watermarks.

->
Learning prompt

An LLM extracts only high-signal personal facts and recurring patterns.

Filtering

Low-importance facts are skipped. Generic world facts are explicitly discouraged.

->
Belief writes

Accepted facts are stored via remember() with origin inferred and learning-run provenance.

Watermarks

Per-domain watermarks prevent reprocessing the same signals repeatedly.

->
Slow accumulation

Knowledge can eventually influence beliefs, but only through a selective distillation path.

9. Telegram push and companion commands

Ready brief

A ready briefing with no Telegram sent marker is picked up by the push loop.

->
Chat target resolution

Push logic resolves thread-linked chats, Program-linked chats, or the owner chat for daily briefs.

Delivery format

Standard briefs become concise HTML summaries. Report briefs can also send visuals and rendered documents.

->
Companion commands

Telegram users can list briefs, list Programs, reply, create actions, complete actions, and correct beliefs using the shared backend model.

quality gates

Validation, harness, and operational loops

The repo is opinionated about shipping behavior with proof. Product loop changes are supposed to be backed by tests, harness runs, checklists, and evidence packs.

Build and type safety

Standard validation stack includes pnpm test, pnpm lint, pnpm typecheck, and pnpm build.

Behavior harness

Core-loop changes are expected to use pnpm harness:core-loop. Repo-wide regression integrity uses pnpm harness:regressions.

Evidence model

Non-trivial tasks are expected to carry a task contract, evidence pack, and at least one matching checklist in harness/checklists.

Operational loops running in the server

  • Periodic daily briefing queueing
  • Due Program schedule polling
  • Background learning from recent signals
  • Knowledge TTL cleanup
  • Artifact cleanup
  • Telemetry cleanup

Measurement loops

  • Product events distinguish user loop events from raw telemetry spans.
  • Examples: Program created, brief follow-up asked, brief action created/completed, belief corrected.
  • The goal is to measure whether the recurring decision loop is actually being used and improved.

product discipline

What the project is optimizing for, and what it is not

This codebase has a lot of capability, but the design discipline is to keep the recurring decision loop at the center and avoid letting internal execution nouns dominate the story.

Primary center

  • Programs and Briefs are the dominant product objects.
  • Tracked follow-through is optional, not a full task-manager strategy.
  • Beliefs and knowledge are trust infrastructure for the next brief.
  • Telegram is a companion surface, not a separate control plane.

Things intentionally kept secondary

  • Threads are interaction containers, not the main product identity.
  • Jobs, schedules, and swarm are implementation objects, not the product story.
  • Search, browser automation, and sandbox are optional enrichments, not correctness dependencies.
  • Knowledge is not the same thing as memory; long-form source material stays separate from distilled beliefs.

The shortest accurate summary

pai keeps track of ongoing decisions in the background, delivers recommendation-first briefs, remembers durable user context, and lets the user correct the system so later briefs improve. Everything else in the repo either supports that loop directly or is being constrained so it does not overshadow it.
